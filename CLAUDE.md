# つづく（tsuduku）
仲間と続ける運動ログ共有アプリのプロトタイプ。次セッション以降もこの方針を維持すること。

## プロダクトの魂（憲法の最上位・開発者本人の言葉）
このプロジェクトの**最上位の憲法**。機能を足す・迷う前に必ずここに立ち返る。以下の「開発者の言葉」は**原文のまま**（要約・言い換え・AIの言葉への置換を禁止）。仕様は変わっても、この魂は変えない。

### 開発者の言葉（原文）
> このアプリは、一人だけだと可視化できない、たまに挫折したくなる心を支えるアプリです。それは自分自身も毎日筋トレしているからこそ感じる、「今日はめんどくさい」「さぼってもいいかな、こんなに頑張ったし」という感情から起因します。
>
> ただ、さぼりを「休み」に、頑張りへの報酬を「みんなへの共有」という健全な承認欲求へと変換します。SNSのような不特定多数の人へ自分の体を共有するのではなく、ただ、大切な人、友人、一緒に頑張りたいと思える人に届けるアプリです。
>
> そして自分の頑張りや積み重ねは記録・タイムラインとして永続的に積み重なっていき、最終的に自分の体を1ヶ月前、半年前という風に閲覧して、変化を互いに楽しむことができます。
>
> アプリの設計はとにかくその人を責めない。デュオリンゴにしても他の運動アプリにしても「だめだよ」「まだやらないの?」というような通知が来て、逆に開きたくなくなる。このアプリは自分だけでなく他の人の頑張りも見えるので、自然と「どんな感じかな」と見られる。運動開始は、他の人がやっていると自分もやりたくなる。だから優しい通知により、相手が開始して、自分も帰ってからタイムラインで互いに共有できるアプリにしたい。
>
> これは予定管理アプリではないので、他の予定アプリと目的が被ることはない。
>
> 作った目的の一つは、お父さんやお母さんの運動を一緒にやりたい、一体感が欲しいと思ったこと。将来このフィットネス界隈で、ただ個人事業を開くよりも、周囲のコミュニティとのつながりを重要視していきたい。
>
> 気軽でシンプルで使いやすいアプリを第一に、みんなに愛される絵日記を作り上げたい。

### 基本の流れ
自分の予定を入れる → それはどのグループから見ても同じ(同期) → 予定画面では当日運動予定の人の予定が見える → 運動開始はタイマーとともに、終われば記録として積み上がる

### この魂から導かれる原則（既存の設計判断との対応）
- **通知は催促ではなく「共有」と「誘い」**。「まだやらないの?」の類は**絶対に作らない**。→ 対応: 優しい通知＝相手のトレ開始で「自分もやりたくなる」誘い／いいね・🔥・コメントはアプリ内🔔に静かに蓄積(Phase 4d)。デュオリンゴ型の催促・詰め通知は禁止。オンボーディングの文言もこの「声」に従う。
- **責めない・晒さない・煽らない（ノーシェイム）**。→ 対応: 未達成は静かに流す(×も通知も残さない)／赤で煽らない・達成率リング廃止／迷ったら**「これは監視か、応援か」**で決める(チートデイ廃止もこの判断＝罪悪感の器を作らない)。「誰が何回サボった/飲んだ」を数える監視機能は作らない。
- **さぼりを「休み」に変換する**。→ 対応: 休養は独立ステータス(type='rest')で運動集計から除外し、**宣言して休んだ日はストリークを維持(切れない)＝計画の一部**(「設計判断・確定」節に記録)。休養は肯定的に🌙で表示する。
- **共有範囲は「一緒に頑張りたい人」だけ**（不特定多数に体を晒さない）。→ 対応: つながり型で `is_connected` をRLSにDBレベル強制／他人に見せる情報は public_profiles ビュー経由のみ／生の体重・体脂肪は本人のみ。大人数SNS化・バズ・拡散は狙わない(密な小人数が核)。**これがスペース隔離とRLSを敷く理由**。
- **記録は永続的に積み上がる**（1ヶ月前・半年前の自分と比べて変化を互いに楽しむ）。→ 対応: 本質は「運動・記録の絵日記」。entries/posts は owner に紐づき永続／記録(のび)画面で可視化。積み上げを消さない。
- **予定管理アプリではない**（他の予定アプリと目的が被らない）。→ 対応: 画面の役割＝予定は「宣言・入力」、記録は「積み上げの閲覧」。カレンダーはあるが目的は**継続支援**であって予定管理ではない。ここを混同した機能は作らない。
- **気軽・シンプル・愛される絵日記を第一に**。→ 対応: デザインはライト/ミニマル/teal 1アクセント/ヘアライン／みんなが慣れたUIパターンを優先(独自を発明しない)＝学習コストを下げ「愛される」に効かせる。

### 起動時の義務
- セッション開始時は必ず **CLAUDE.md 全文と TODO.md を読んでから**作業に入る。要約・一部読みで動き出さない。

### 絶対規則（モデルの判断で省略・変更してはならない）
1. **差分駆動**: コード変更は必ず「差分概要を提示 → 人間のOK → 適用 → コミット」。いきなり適用しない。大きい変更は分割案を出す。
2. **SQL/RLS/スキーマ変更**: 実行する SQL は必ず**全文をチャットに提示**し、人間が確認してから **Supabase で人間が実行**する。要約や説明だけで実行させない。
3. **push 前に人間のブラウザ確認**を挟む（合格ライン＝検証方法を明示して確認してもらう）。
4. **確定した設計判断は再議論しない**。便利そうでも蒸し返さない（例: タイムライン投稿は当日実施のみ＝盛れない構造／生の体重・体脂肪は本人のみ／ノーシェイム＝未達を晒さない／チートデイ廃止）。変更したい時は人間に経緯を示した上で**明示的に提案**する。
5. **技術選定の経緯を尊重する**: 「なぜ」が記録されているものを新しい流儀で勝手に置き換えない（例: Tailwind v3 固定＝v4 に上げない／id はクライアント生成 `crypto.randomUUID()`／`.insert().select()` の RLS 循環回避パターン／GRANT と RLS は両方必要）。

### 運用の暗黙知（明文化）
- **対話は日本語**。ユーザーは非エンジニアだが判断力が高い。専門用語は噛み砕き、判断ポイントは選択肢で示す。
- **各セッション終了時に、次回起動用の「文脈取り戻しプロンプト」を提示**する。
- **コミットは意味のある単位で分割**し、TODO.md / CLAUDE.md の進捗更新を忘れない。
- **不明点・設計ギャップは勝手に埋めず人間に確認**する（今までもそれで品質を保ってきた）。
- **オンボーディングツアーの構成・文言はユーザー(開発者本人)が細かく指示する領域**。AI側から説明文の追加・拡充を提案・実装しない。ツアーはアプリの「声」でありプロダクトオーナーが直接管理する（指定文言は一字一句そのまま表示・AIがタイトル等を足さない）。

## ビジョン
このアプリは「運動・記録の絵日記」。ずっと続いて、身内・自分・友達で高め合い、長く愛されることを目指す。SNS的だが承認欲求ではなく継続支援が核。機能追加時は常にこの軸（続けたくなる／愛される／高め合う）に照らして判断する。

### プロダクトの魂（なぜ作るか）
→ **冒頭「プロダクトの魂（憲法の最上位・開発者本人の言葉）」を参照**（原文と、そこから導かれる原則＝ノーシェイム／監視ではなく応援／さぼりを休みに／共有は一緒に頑張りたい人だけ、が最上位の憲法）。

### 設計の憲法（破ってはいけないもの — 詳細は「設計原則(厳守)」「最優先のプライバシー原則」節）
- **スペース完全独立・隔離**。あるスペースの記録/投稿が別スペースに漏れない。RLSでDBレベル強制（アプリがバグってもDBが門番）。
- **生の体重/体脂肪は本人のみ**。どのスペースにも生数値を出さない。共有は前週比など変換値のみ。
- **画面の役割を混ぜない**。予定=入力／タイムライン=閲覧+リアクション／記録=閲覧(可視化)。入力系UIは原則「予定」に置く。
- **デザインはライト/ミニマル/teal 1アクセント/ヘアライン**。黒背景・ネオン・近未来は不採用。
- **ユーザーの表示カスタムを尊重**（ウィジェット式ON/OFFで要らない情報は隠せる方向。押し付けない）。
- **みんなが慣れているUIパターンを優先する**（独自を発明せず、既存アプリの定番に寄せる。例=スペース切替はDiscord/Slack型、投稿写真は撮影/アルバムのOS標準、引っ張って更新はInstagram型、初回はコーチマークツアー）。学習コストを下げ「愛される」に効く。

### 技術判断の記録（なぜこうしたか）
- **スタック**: Vite + Vanilla JS（フレームワークなし）/ Supabase（Auth + Postgres + RLS）/ Vercel。軽さと「DBが門番」を両立するため。Vanilla維持＝プロトタイプの見通しと学習コストを優先。
- **GitHub Pages → Vercel へ移行した理由**: Supabaseキーを環境変数としてビルドに注入する必要があり、静的Pagesでは秘匿configを扱いにくい。Vercelは env とプッシュ自動デプロイが素直。
- **id はクライアント生成（`crypto.randomUUID()`）にした理由**: 挿入時にDB採番idへ付け替える手間が無く、ローカル即描画とDB行を同一idで一致させられる。さらに下記 `.insert().select()` のRLS循環も回避できる。
- **GRANT と RLS は両方必要（403 permission denied の解決経緯）**: RLSを有効化しても `authenticated` ロールにテーブルGRANTが無いと `42501 permission denied`(403)で全滅する。schema.sql末尾で authenticated にのみ CRUD/EXECUTE を付与（anonには一切付与しない=ログインゲート＋プライバシー）。実際に403で詰まり、これで解決した。
- **`.insert().select()` のRLS循環（BUG A）と回避**: spaces を作った直後に読み返すと、まだ space_members 行が無く `is_space_member()=false` で0行→`PGRST116`。回避＝idをクライアント採番して読み返し自体を廃止し、membershipを別途insertする。読み返しが安全なのは、自分の行が自分のSELECTポリシーで見えるテーブル（例 users_select_self）だけ。
- **日付はハードコード（`TODAY` 固定・週ストリップも固定）にした経緯**: 実日付連動は週ストリップ/月カレンダー/曜日色付けの全面改修になるため、記録永続化とは切り離して独立タスク化（TODO.mdバックログ）。今やると二度手間になる判断。
- **onAuthStateChange 内で直接 `await supabase.from(...)` しない**: GoTrueのロックでデッドロックし得る。DB処理は `setTimeout(0)` 等でコールバックの外に出す（現状未修正・書き込み追加時に対応）。
- **bootstrap(ログイン時=INSERT時)にのみ走る処理は、既存セッション/既存行に反映されない**: 例＝users.photo を「行が無いとき」だけ入れると、機能追加前に作られた既存ユーザーには永遠に入らない（再ログインしても行は在るのでINSERTは走らない）。**セッション継続中にも効かせたい既定値は、画面表示時/ロード時の同期フォールバックを併設する**（photoは bootstrap で「行はあるが空なら UPDATE で遡り補完」を追加して解決）。
- **ALTERは実行フローに乗せ、実際に確認するまで「完了」扱いにしない**: 提示しただけ/実行したつもりで進めると、後段で `42703 column ... does not exist` になる（tour_doneで発生）。列追加は「全文提示→人間が実行→REST/Table Editorで存在確認」まででワンセット。
- **DB構造変更(ALTER)後はハードリロードしてから検証する**: Supabaseの接続/スキーマキャッシュ(PostgREST)が古いままだと、追加済みの列でも `column does not exist` が出る。ALTER後はハードリロード(接続・スキーマキャッシュの張り直し)してから動作確認する。

## コンセプト
- 仲間と一緒に運動を「続ける」ことが核。SNS的な運動ログ共有アプリ
- 予定の「宣言」が核心機能: きょう誰が何をするかを先に宣言する=サボれない仕組み

## 3画面
- 記録(Feed): 仲間の運動投稿。部位タグ・実施データ(実施時間/消費/摂取/カロリー差分)・リアクション(🔥💪👏)
- 予定(Schedule): 週ストリップ(**日曜始まりに確定**・< >で先週/来週ナビ・「今日」で復帰)、きょうの宣言、予定リスト(予定/実施中/実施済/予定変更/未実施)。週境界(週ストリップ/月カレンダー/週次集計)はすべて日曜始まりで統一。日付は実日付(端末ローカル)連動。※予定達成率リング(%)は廃止＝プレッシャーになるため(ノーシェイム)。
- のび(Progress): 自分ファースト。運動日数/連続記録/体重を大きく、体重折れ線(週/月)、運動日数棒、仲間のうごきは最下部に小さく

## 画面の役割
- 予定 = 入力する場所。運動・体重・食事などの記録入力はここに集約する
- タイムライン = 閲覧＋リアクション
- 記録 = 閲覧（可視化）
- 今後の入力系UIは原則「予定」に置く
- 予定の各エントリ(運動/体重/食事)はタップで編集・削除できる(logEntriesにidを持たせて特定。過去日も週/月から修正可)。体重・食事の編集は記録グラフへ即反映
- 月カレンダーのインライン詳細は「その日時点」で出し分ける: 過去日=その日の記録(運動/体重/食事の実績)、当日・未来=予定(誰が何をする)

## Schedule拡張仕様
Scheduleは「これからやること(do)＋これから守ること(commitment)を小人数で共有するハブ」。カレンダー項目は type を持つ:
- workout: 部位タグ(胸/背中/脚/肩/腕/有酸素/ストレッチ＋「＋その他」で自由テキスト追加可)は複数選択可(データは tags 配列)・時間・ステータス(予定/実施済/未実施/予定変更)。複数時は複数チップ/複数ドットで表示。DBの tags は text[] なので自由テキストも変更不要で保存できる。**新規予定は部位を初期選択しない**(未選択スタート・部位なしでも「運動」として記録可=必須にしない)。「＋その他」は**シートを開かずインライン入力**(チップ列直下に 入力＋追加＋× を展開)。
- rest(休養)【独立ステータス・type='rest'】: 休養は"部位"ではなく運動と別のステータス。予定シートの🌙休養トグル(部位と排他=選ぶと部位が全外れ)で作る。落ち着いたスレート#8993A8＋🌙「休養日」で表示(運動の緑とは別トーン)。**運動日数/運動時間の集計には含めない**(実績ではない)。タイマー(運動開始)は休養日には出さない(気が変われば予定に部位を足せば復活)。ノーシェイム=サボりではなく計画的回復として肯定的に表示。RLSは運動宣言と同じく「つながり」に共有(体重/食事は本人のみ)。
- meal(本日の食事): 摂取kcalのみをざっくり入力(食品DB検索なし・P/F/Cは廃止=毎食入力が重いため)。摂取kcalは記録のカロリー収支グラフ(週)の実データ源(カロリー差分)。月はmock
- limit(自分ルール)【日数ストリーク型・Day3-B確定 / Snapchat型】: 例「甘いものを食べない」「揚げ物我慢」。公開(みんなが見れる)/自分だけ(非公開)の2択・最大3つ。**追加日から毎日そっと+1**(実日付の引き算=`streak_start`から今日までの経過日数+1・開始日=1日目)。破らない限り**操作は要らない**。**自己ベストは持たない**(fit tree=育てる思想・過去と比較させない。DBの`streak_best`列は残置・未使用)。操作は**✕→軽い2択**のみ=「🌱また1日目から育てる」(streak_startを今日にリセット)/「このルールを終える」(削除)、どちらもノーシェイム。破っても晒さない・投稿しない・責めない。行はミニマル(ルール名フル＋🔥N日目＋公開/非公開マーク＋✕)。「🔥有言実行 N連続」の集約表示と達成カード(kind:'achieve')は廃止(タイムラインはシンプルに)。金銭/ポイント系は作らない。
- ~~resist_log(我慢ログ/踏みとどまった)~~: **Phase 3b-2で完全削除**(アイコンの意味が伝わらない・毎日入力の主役は通常のトレ投稿にする、という判断)。kind:'resist'投稿・UI・ロジックすべて撤去済み。

### 設計原則(厳守)
- ノーシェイム: 破っても赤や警告で出さない。色は警告ではなく示唆に使う
- 項目別プライバシー: 各データに公開範囲(全員/グループ内/特定の相手だけ/自分だけ)。体重・体脂肪は「数値そのものは非公開、前週比/変化率/達成率だけ共有」モードを用意。写真は投稿ごとに範囲選択
- お酒等は「誰が何回飲んだか」を晒すカウンターにしない
- 監視ではなく前向きな共有・称賛を主役に
- （状態）バックエンド/認証/保存は Phase 2–3 で実装済み。以降の新データも同じ RLS + owner/space_id の型に載せる

## アーキテクチャ確定版：つながり型（唯一の正・2026-07 確定）
これが共有・可視性の**唯一の正**。以前の「スペース(箱)＝タイムラインの単位」型は破棄。以降この定義に反する実装・提案をしない。

### 定義(5項目)
1. **予定・記録(entries)は所有者(owner)にのみ紐づく**。どのグループから見ても本人の予定は常に同一(箱ごとに予定が変わることはない)。
2. **タイムラインは1本**。自分＋つながっている全ユーザーの投稿(posts)が時系列で流れる。投稿先の選択・タイムラインの切替は存在しない。
3. **spaces は「グループ＝つながる単位」として再定義**。グループ参加=そのメンバー全員とつながる。招待コード(4b)はグループ招待。複数グループ所属可。「つながっている」=いずれかのグループを共有している。
4. **プライバシーは2段のみ**。本人のみ(生の体重/体脂肪/非公開ルール/設定) / つながった人(公開プロフィール・投稿・公開ルール・共有用の変換値)。
5. **UIにスペース切替・投稿先選択・「◯◯のタイムライン」表記は作らない**(Phase 5計画からも削除)。「メンバーを招待」はグループ招待として維持。

### 変更経緯(なぜ箱型をやめたか)
- 開発者本人の設計判断: **「予定は"体に紐づく事実"であり、箱(スペース)で変わるものではない」**。同じ人の同じ予定が、見るグループによって違って見えるのは不自然。so entries は owner に固定し、可視性だけを「つながり」で制御する。
- 箱型(スペースごとに別タイムライン・投稿先選択)はカップル/親友2-5人の密な絵日記には過剰で、切替の認知コストが「気軽さ」を損なう。1本のタイムライン＝慣れたSNSパターン(設計憲法)。

### 技術メモ(実装済み・Step 1-3)
- **可視性は space_id ではなく owner とのつながりで判定**。`is_connected(p_user)` = 自分自身 or いずれかのグループ(space_members)を共有 → true(自分も常にtrue)。SECURITY DEFINERで再帰回避。
- entries/posts の select = `is_connected(owner)`、rules = `owner=自分 or (pub and is_connected(owner))`。書き込みは owner 本人のみ。**space_id列は残置(Option A)＝書き込みの門番から外れ、可視性判定にも使わない**(vestigial)。
- **他人に見せる情報は public_profiles ビュー経由のみ**(nickname/photo)。users直読みは本人のみ。生体重等は絶対に露出しない。
- **グループ参加は招待コードのみ**: `invitations`(コード=hex12・既定24h・人数無制限) + `join_space_with_code()`(唯一の入口・SECURITY DEFINER・anon実行不可)。直接の space_members 自己追加は「自分が作成したグループのみ」に厳格化。
- アプリ: loadAll は entries/rules=本人・posts=つながり(owner非限定=1本TL)。招待/参加UIは設定シートの「つながる(グループ)」欄。
- バッチ3でダミーユーザー/サンプル記録を全削除し、自分(ぼーい)一人・記録ゼロのフラット初期状態に。データ構造(logEntries/posts/limits等)は温存し中身だけ空。グラフ/一覧/ヒーローは全てlogEntriesから算出、データなしはノーシェイム空状態(🌱)で表示。Supabase接続時はこの空構造にwho/公開範囲を載せて実データを流す。

## データガバナンス方針（規約化前の骨子）
利用規約・プライバシーポリシーを作る前の設計骨子。実装・文書化はまだ（TODO参照）。家族以外を入れる前(Phase 4)に規約化する。

### 規約の基本方針
- 一般的な利用規約・プライバシーポリシーの構成をベースに、**データ利用目的を「具体的に列挙して広くカバーする」方式**を採用する。
- 理由: 健康関連データ(体重・体脂肪・食事・運動)は要配慮情報に近く、包括的な「幅広く運用可能」という書き方は法的に機能しない。**目的を列挙することで実質的な広さを適法に確保**する。きちんと同意を取ったデータだけが将来の事業価値(提携・売却時のDD)を持つ。

### 利用目的の列挙（規約に明記する項目）
1. サービスの提供・維持・改善・不具合対応
2. 匿名化・統計化した上でのデータ分析、研究機関・事業パートナーへの提供(個人を特定しない形・本人同意ベース)
3. 新機能・関連サービスの開発への活用
4. 広告配信の最適化(ユーザーデータを外部広告事業者に渡さない範囲)
5. 法令に基づく開示要請への対応、および安全上の重大な問題(事件性のある事案等)への対応としての管理者アクセス
6. 事業譲渡・合併等の際のデータ承継

### 管理者アクセスの原則
- 運営者(DB管理者)は技術上、全データ(将来的には写真含む)にアクセス可能(service_role/Table Editor の上位権限)。
- ただし行使は「法令に基づく要請・重大な安全上の問題・本人の同意がある場合」に限定し、**日常的な閲覧はしない**(スペース隔離・プライバシー死守の憲法と両立)。
- この条件を利用規約/プライバシーポリシーに明記することが、運営者自身の法的保護にもなる。

### 機密情報の非保持
- クレジットカード情報・パスワード等の機密情報の入力欄は作らない(設計レベルで排除)。
- 将来の課金は決済代行(Stripe等)に完全委任し、カード情報を自前で保持しない。
- 規約に「当サービスはクレジットカード情報を保持しません」と明記する。

### 広告の方針
- 無料提供の収益補助として広告掲載の可能性を規約に明記。
- ユーザーデータを広告事業者に渡さない**単純広告**から始める(ターゲティング広告はプライバシーポリシー要件が重くなるため当面回避)。
- 親密な空間(タイムライン内)への挿入は避け、場所を厳選(記録画面下部等)。

## バックエンド設計(Supabaseフェーズ)
次フェーズの設計メモ。実装はまだ。着手時はこの方針に沿う。

### 採用モデル: つながり型（旧「スペース(箱)型」は破棄・上記「アーキテクチャ確定版」が正）
- spaces は「グループ＝つながる単位」。グループ参加=そのメンバー全員とつながる。複数グループ所属可。一人でも使える(自分のグループ)。
- 予定・記録は owner に紐づき、どのグループから見ても同一。タイムラインは1本(自分＋つながり)。**投稿先選択・スペース切替・「◯◯のタイムライン」表記は作らない**。
- グループ参加は招待コードのみ(join_space_with_code が唯一の入口)。「メンバーを招待」はグループ招待として維持。

### 最優先のプライバシー原則(絶対に破らない)
- **可視性は「つながっているか(is_connected)」で判定**。つながっていない人同士は互いのデータを一切見られない。これをRLSでDBレベルに強制(アプリがバグってもDBが門番)。
- 生の体重・体脂肪・体型・非公開ルール・設定は**本人だけ**。共有は前週比など変換値、または public_profiles ビュー(nickname/photo)のみ。生数値はどこにも出さない。
- 他人に見せる情報は public_profiles ビュー経由のみ。users テーブルの直読みは本人限定を維持。

### テーブル設計(v1)
- users: id/nickname/写真/身長/体重/体脂肪率/メンテカロリー(生数値は本人のみ)
- spaces: id/名前(変更可)/アイコン画像(変更可)/作成者/作成日
- space_members: space_id/user_id/役割。「同じスペースにいるか」でRLS判定
- entries(記録): id/owner/space_id/type(workout/weight/meal)/日付/tags/実施時間/値/status
- posts(投稿): id/owner/space_id/kind(workout/resist/achieve)/写真/コメント/リアクション
- rules(自分ルール): id/owner/内容/pub(公開/自分だけ)/space_id

### RLS方針（つながり型・Step 1で書き換え済み）
- entries/posts: **`is_connected(owner)`** の行だけ閲覧可(自分＋つながっている人)。つながっていない人の行は一切不可。space_id では絞らない。
- rules: 本人は全部 / つながっている人には公開(pub=true)のみ。非公開は本人だけ。
- usersの生の体重/体脂肪/身長/メンテ/設定: 本人のみ(users_select_self)。共有は変換値 or public_profiles のみ。
- 記録の作成/編集/削除: owner本人のみ(space非依存)。
- **他人に見せる情報は `public_profiles` ビュー経由のみ**。users直読みは本人限定。nickname/photo 等の最小限だけを露出し、生体重/体脂肪/身長/メンテ/設定は絶対に含めない・anon非公開。
- グループ参加は `join_space_with_code()`(唯一の入口・SECURITY DEFINER・anon実行不可)。直接の space_members 自己追加は「自分が作成したグループのみ」。
