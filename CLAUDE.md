# つづく（tsuduku）
仲間と続ける運動ログ共有アプリのプロトタイプ。次セッション以降もこの方針を維持すること。

## ビジョン
このアプリは「運動・記録の絵日記」。ずっと続いて、身内・自分・友達で高め合い、長く愛されることを目指す。SNS的だが承認欲求ではなく継続支援が核。機能追加時は常にこの軸（続けたくなる／愛される／高め合う）に照らして判断する。

## 思想継承（引き継ぎ者へ — 別のAI/開発者が同じ判断基準で続けるために）
このプロジェクトを引き継ぐ人（人間でもAIでも）は、機能を足す前にこの節を読むこと。個々の仕様は変わっても、以下の「なぜ」は変えない。詳細な各論は後続の各節にある。

### プロダクトの魂（なぜ作るか）
- 本質は「運動・記録の絵日記」。カップル・親友2〜5人の密な関係で、そっと高め合い、長く愛されること。大人数SNS・バズ・拡散は狙わない。
- **ノーシェイム最優先**。失敗を罰しない・晒さない・赤で煽らない。未達成は静かに流す（×も通知も残さない）。承認欲求ではなく「継続支援」が核。
- **監視アプリに転ばせない**。「誰が何回サボった/飲んだ」を数える機能は作らない。チートデイを廃止したのもこの判断（罪悪感の器になるため）。迷ったら「これは監視か、応援か」で決める。

### 設計の憲法（破ってはいけないもの — 詳細は「設計原則(厳守)」「最優先のプライバシー原則」節）
- **スペース完全独立・隔離**。あるスペースの記録/投稿が別スペースに漏れない。RLSでDBレベル強制（アプリがバグってもDBが門番）。
- **生の体重/体脂肪は本人のみ**。どのスペースにも生数値を出さない。共有は前週比など変換値のみ。
- **画面の役割を混ぜない**。予定=入力／タイムライン=閲覧+リアクション／記録=閲覧(可視化)。入力系UIは原則「予定」に置く。
- **デザインはライト/ミニマル/teal 1アクセント/ヘアライン**。黒背景・ネオン・近未来は不採用。
- **ユーザーの表示カスタムを尊重**（ウィジェット式ON/OFFで要らない情報は隠せる方向。押し付けない）。

### 技術判断の記録（なぜこうしたか）
- **スタック**: Vite + Vanilla JS（フレームワークなし）/ Supabase（Auth + Postgres + RLS）/ Vercel。軽さと「DBが門番」を両立するため。Vanilla維持＝プロトタイプの見通しと学習コストを優先。
- **GitHub Pages → Vercel へ移行した理由**: Supabaseキーを環境変数としてビルドに注入する必要があり、静的Pagesでは秘匿configを扱いにくい。Vercelは env とプッシュ自動デプロイが素直。
- **id はクライアント生成（`crypto.randomUUID()`）にした理由**: 挿入時にDB採番idへ付け替える手間が無く、ローカル即描画とDB行を同一idで一致させられる。さらに下記 `.insert().select()` のRLS循環も回避できる。
- **GRANT と RLS は両方必要（403 permission denied の解決経緯）**: RLSを有効化しても `authenticated` ロールにテーブルGRANTが無いと `42501 permission denied`(403)で全滅する。schema.sql末尾で authenticated にのみ CRUD/EXECUTE を付与（anonには一切付与しない=ログインゲート＋プライバシー）。実際に403で詰まり、これで解決した。
- **`.insert().select()` のRLS循環（BUG A）と回避**: spaces を作った直後に読み返すと、まだ space_members 行が無く `is_space_member()=false` で0行→`PGRST116`。回避＝idをクライアント採番して読み返し自体を廃止し、membershipを別途insertする。読み返しが安全なのは、自分の行が自分のSELECTポリシーで見えるテーブル（例 users_select_self）だけ。
- **日付はハードコード（`TODAY` 固定・週ストリップも固定）にした経緯**: 実日付連動は週ストリップ/月カレンダー/曜日色付けの全面改修になるため、記録永続化とは切り離して独立タスク化（TODO.mdバックログ）。今やると二度手間になる判断。
- **onAuthStateChange 内で直接 `await supabase.from(...)` しない**: GoTrueのロックでデッドロックし得る。DB処理は `setTimeout(0)` 等でコールバックの外に出す（現状未修正・書き込み追加時に対応）。

### 開発の進め方（このプロジェクトの流儀）
- **差分駆動**: まず差分概要を出す → 人間のOK → 適用 → コミット。大きければ分割案を出す。いきなり大量に書かない。
- **データ/RLS/SQLに触る変更は、実行前に必ず全文を人間が確認**。SQLは人間がSupabaseで実行し、AI側は検証を担う（例: anonプローブで漏れが無いか確認）。
- **各フェーズに合格ライン（検証方法）を定義してから進む**。例: 3a=「ログイン→3テーブル各1行／プロフィール保存→リロード復元」を実機で通す。
- **UIの細部は機能が固まってから一括調整**（構造が変わる前に磨くと二度手間）。気づいた見た目改善はTODO.mdバックログに記録して先送り。
- **CLAUDE.md / TODO.md をソース・オブ・トゥルースとして常に更新**。学び（失敗の解決経緯を含む）を必ず残す。

## コンセプト
- 仲間と一緒に運動を「続ける」ことが核。SNS的な運動ログ共有アプリ
- 予定の「宣言」が核心機能: きょう誰が何をするかを先に宣言する=サボれない仕組み

## 3画面
- 記録(Feed): 仲間の運動投稿。部位タグ・実施データ(実施時間/消費/摂取/メンテ差分)・リアクション(🔥💪👏)
- 予定(Schedule): 週ストリップ、今週の予定達成率リング、きょうの宣言、予定リスト(実施済/予定変更/未実施)
- のび(Progress): 自分ファースト。運動日数/連続記録/体重を大きく、体重折れ線(週/月)、運動日数棒、仲間のうごきは最下部に小さく

## 画面の役割
- 予定 = 入力する場所。運動・体重・食事などの記録入力はここに集約する
- タイムライン = 閲覧＋リアクション
- 記録 = 閲覧（可視化）
- 今後の入力系UIは原則「予定」に置く
- 予定の各エントリ(運動/体重/食事)はタップで編集・削除できる(logEntriesにidを持たせて特定。過去日も週/月から修正可)。体重・食事の編集は記録グラフへ即反映
- 月カレンダーのインライン詳細は「その日時点」で出し分ける: 過去日=その日の記録(運動/体重/食事の実績)、当日・未来=予定(誰が何をする)

## デザイン方針(あすけん/hacomono系のミニマル)
- アクセントはteal #14B87C の1色。白基調 bg#FAFAF8 / ink#181D1A / sub#6E736C
- ヘアライン主体・余白広め・数字は大きく。アバターは頭文字の単色円
- 部位タグ色: 胸#FF6A3D 背中#3E86C9 脚#7C6CD0 肩#E0A53A 腕#B5836A 有酸素#14B87C ストレッチ#5FB6A8 休養#9AA09A（肩腕は肩/腕に分割）。ユーザーが「＋その他」で追加する自由テキストタグは色ニュートラル灰#9AA09A固定（tagDotのフォールバック）

## やらないこと
- 黒背景・ネオン・近未来テーマは不採用(ミニマル方向と喧嘩する)
- 大量拡散・バズ・大人数SNS化は狙わない(密な小人数が核)。PWA化はPhase 6。
- ※歴史的記述の訂正: 当初は「単一HTMLの静的プロトタイプ／localStorage」だったが、Phase 1でVite化・Phase 2以降Supabase接続済み。正はクラウド(Supabase)。

## Schedule拡張仕様
Scheduleは「これからやること(do)＋これから守ること(commitment)を小人数で共有するハブ」。カレンダー項目は type を持つ:
- workout: 部位タグ(胸/背中/脚/肩/腕/有酸素/ストレッチ/休養＋「＋その他」で自由テキスト追加可)は複数選択可(データは tags 配列)・時間・ステータス(予定/実施済/未実施/予定変更)。複数時は複数チップ/複数ドットで表示。DBの tags は text[] なので自由テキストも変更不要で保存できる
- meal(本日の食事): 摂取kcalのみをざっくり入力(食品DB検索なし・P/F/Cは廃止=毎食入力が重いため)。摂取kcalは記録のカロリー収支グラフ(週)の実データ源(メンテ差分)。月はmock
- limit(自分ルール)【週次振り返り型・Phase 3b-2】: 例「飲みは週2まで」「ラーメン我慢」。公開(みんなが見れる)/自分だけ(非公開)の2択・最大3つ。**週1回「今週守れた」をタップするだけ**(守れた✓のみ・守れなかったは何も記録しない=ノーシェイム)。守れた→**連続週数(streak)+1・自己ベスト(streak_best)更新**。公開ルールならFeedに達成カード(kind:'achieve')を投稿=カード内に「🔥N週連続(自己ベストM週)」を併記し友達にも連続とベスト更新が見える。非公開は静かに+1のみ。報酬は連続週数(自己ベスト併記)のみ・金銭/ポイント系は作らない。週リセット(週フラグ解除)は実日付化とセットの将来タスク。
- ~~resist_log(我慢ログ/踏みとどまった)~~: **Phase 3b-2で完全削除**(アイコンの意味が伝わらない・毎日入力の主役は通常のトレ投稿にする、という判断)。kind:'resist'投稿・UI・ロジックすべて撤去済み。

### 設計原則(厳守)
- ノーシェイム: 破っても赤や警告で出さない。色は警告ではなく示唆に使う
- 項目別プライバシー: 各データに公開範囲(全員/グループ内/特定の相手だけ/自分だけ)。体重・体脂肪は「数値そのものは非公開、前週比/変化率/達成率だけ共有」モードを用意。写真は投稿ごとに範囲選択
- お酒等は「誰が何回飲んだか」を晒すカウンターにしない
- 監視ではなく前向きな共有・称賛を主役に
- （状態）バックエンド/認証/保存は Phase 2–3 で実装済み。以降の新データも同じ RLS + owner/space_id の型に載せる

## 共有モデル（次フェーズ方針・バックエンド必須）
- 「グループの箱」型ではなく「自分中心＋相手ごとの公開範囲」型を採用する。
  理由: カップル/親友2-5人の密な関係が核、項目別プライバシーと両立、絵日記的に気軽に始められる。
- 自分のアカウントが予定/記録/タイムラインを持ち、友達を個別に追加。各データは「項目ごと・相手ごと」に公開範囲を選べる（既存の項目別プライバシー設計を踏襲）。よく一緒にやる数人は「仲間」としてビューでまとめる（箱型グループではなくビュー）。予定には自分が見せたい人を個別に追加できる。
- 現在のmockデータ構造は、将来この共有モデルに移行できるよう logEntries に who(所有者)と公開範囲フィールドを持たせる方向で整える（今は単一ユーザーのmockのみ）。
- バッチ3でダミーユーザー/サンプル記録を全削除し、自分(ぼーい)一人・記録ゼロのフラット初期状態に。データ構造(logEntries/posts/limits等)は温存し中身だけ空。グラフ/一覧/ヒーローは全てlogEntriesから算出、データなしはノーシェイム空状態(🌱)で表示。Supabase接続時はこの空構造にwho/公開範囲を載せて実データを流す。

## バックエンド設計(Supabaseフェーズ)
次フェーズの設計メモ。実装はまだ。着手時はこの方針に沿う。

### 採用モデル: スペース(Slack/Discord型)
- ユーザーは複数の「スペース(共有の箱=タイムライン/カレンダーの単位)」を持てる。一人でも使える(自分だけスペース)。文脈ごとに共有先を分ける(夫婦/家族/筋トレ仲間/職場 等)
- スペースは＋で新規作成でき、アイコン画像・名前を後から編集できる(Discord型)。左サイド(スマホは上部)で切り替え
- 投稿・記録は「どのスペースに属するか/出すか」を選べる。投稿時にタイムライン(スペース)を選択する

### 最優先のプライバシー原則(絶対に破らない)
- 各スペースの情報は完全に独立・隔離する。あるスペースの記録/投稿が別スペースに漏れることは絶対にない
- 「同じスペースのメンバー」以外はそのスペースの中身を一切見られない。これをRLS(Row Level Security)でデータベースレベルに強制する。アプリのコードがバグっても、DBが門番として弾く
- 生の体重・体脂肪・体型は本人だけ。どのスペースにも生数値は出さず、共有は前週比など変換値のみ
- スペースをまたいだデータ混在は起こさない。表示も取得も選択中のスペースに閉じる

### テーブル設計(v1)
- users: id/nickname/写真/身長/体重/体脂肪率/メンテカロリー(生数値は本人のみ)
- spaces: id/名前(変更可)/アイコン画像(変更可)/作成者/作成日
- space_members: space_id/user_id/役割。「同じスペースにいるか」でRLS判定
- entries(記録): id/owner/space_id/type(workout/weight/meal)/日付/tags/実施時間/値/status
- posts(投稿): id/owner/space_id/kind(workout/resist/achieve)/写真/コメント/リアクション
- rules(自分ルール): id/owner/内容/pub(公開/自分だけ)/space_id

### RLS方針
- entries/posts: 自分がメンバーであるスペースの行だけ閲覧可。他人のスペースは一切不可
- usersの生の体重/体脂肪: 本人のみ。共有は変換値(前週比)を別途
- 記録の作成/編集/削除: owner本人のみ

### 段階実装計画
- Phase 1: Vite化(土台)
- Phase 2: 認証(Google)＋users。本人情報をクラウド保存
- Phase 3: entriesをSupabase移行。自分だけスペース1つでクラウド保存・復元(共有なし)
- Phase 4: spaces/space_members有効化。共有スペース1つを作り2アカウントでRLSテスト(同じスペースだけ見える)←本丸
- Phase 5: ＋でスペース作成・アイコン/名前編集(Storage)＋複数スペース切り替えUI＋投稿時スペース選択
- Phase 6: リアルタイム同期
- 注: テーブルはPhase1-2の段階から最初からspaces/space_id込みで作る(器を先に用意)。複数スペース切替が動くのはPhase 5

### Phase 3a 完了（実装済み・次は 3b）
Phase 3a 完了・push済み。「ログイン→users行＋自分スペース自動作成→保存データをロードして描画／プロフィールはクラウド保存」までの土台が動作。entries/rules/posts の書き込み配線は 3b。
- **実装**: `src/db.js`(データ層: bootstrap/loadAll/profileFromRow/saveProfileRow) 追加。main.js は initApp を async化(bootstrap→CURRENT_USER/members/profile→loadAll→render→renderIdentity)、CURRENT_USER/members を let化、firstCP()+renderIdentity() 追加、saveProfile でクラウド保存。index.html にヒーロー用 `#heroAvatar`/`#heroName` id 付与。
- **スキーマ要点**: users/spaces/space_members/entries/posts/rules を最初からspace_id込みで作成。RLSは「同スペースのメンバーだけread・owner本人だけwrite」、usersの生体重/体脂肪は本人のみ。space_members参照は無限再帰回避のためSECURITY DEFINERの is_space_member() 経由。プロフィール往復用に users へ `activity_coef`/`maintenance_override` 列を追加(冪等ALTERも同梱)。
- **重要な学び(schema.sql に反映済み)**: (1) **GRANT と RLS は別レイヤー**。RLSを有効化しても `authenticated` にテーブルGRANTが無いと 42501 permission denied で読み書き不可。schema.sql 末尾で `authenticated` にのみ GRANT(anonには一切付与しない=ログインゲート＋プライバシー)。(2) **spaces のinsertで `.select()` 読み返し不可**: membership作成前は is_space_member=false で0行→PGRST116。space idを `crypto.randomUUID()` でクライアント採番し読み返しを廃止(users insertの読み返しは users_select_self でOK)。
- **onAuthStateChange 注意(未修正・要監視)**: 現状は getSession/onAuthStateChange 双方から showApp→initApp。将来 3b で書き込み等を足す際、onAuthStateChangeコールバック内で直に await supabase.from(...) するとGoTrueロックでデッドロックし得る(setTimeout(0)で外に出す)。今回は 42501 が主因だったため未対応。
- **確定した3つの修正**:
  1. 【3bで実施】食事は摂取kcalのみ。entriesから protein/fat/carbs を持たない＋食事入力UIからもP/F/C欄を削除(毎食の細かい入力は重い。将来パッチで写真判定など再検討)。※スキーマ側は既にkcalのみ。残るは confirmSheet の meal 書き込みと入力UIのP/F/C欄削除(entries書き込み配線と同時)
  2. 【3a済】ニックネーム(nick)を表示名・アバター頭文字の唯一の真実に連動。先頭文字は Array.from() でコードポイント単位抽出(str[0]だと絵文字/サロゲートで半欠けするため)。renderIdentity()でヘッダ/記録ヒーローのアバター・名前を更新。初回nick既定値はGoogleプロフィール名。
  3. 【3a済】絵文字対応(ニックネーム等で絵文字が使える。text列＝UTF-8で保存、頭文字抽出もArray.fromで対応)
- **後回し(合意済み)**: 写真の永続化=Phase 5(Storage)・リアクション永続化=後回し・日付モデル(TODAY/週ストリップのハードコード)=現状維持。実装は 3a→3b の2コミットに分割。
- **id方針**: entries/posts/rules のidは crypto.randomUUID() でクライアント生成しDBと同一に(挿入時の付け替え不要)。CURRENT_USERはログイン時に session.user.id を代入(let化)。

### Phase 3b 完了（記録の永続化）
- **3b-1(entries)**: 運動/体重/食事の保存配線(upsertEntry/removeEntry)。編集=id上書き(二重登録なし)・削除=該当行のみ。dur_sec(秒)保存・表示は整形。食事P/F/C削除。部位タグ肩腕→肩/腕分割＋「＋その他」自由テキスト。実績集計はstatus:'done'のみ。本日の体重/食事カードはTODAY選択時のみ表示、過去/未来日は選択日リストにweightRow/mealRowで出す。
- **3b-2(posts + rules)**: 投稿(addPost→upsertPost、タイマー投稿/達成カード)＋自分ルール(upsertRule/removeRule)の保存配線。ルールを**週次振り返り型**に再設計(上記 Schedule拡張仕様参照)。「踏みとどまった」完全削除。**ストリークに自己ベスト併記**(best>currentのときだけ faint 表示。連続が切れてもベストは残す=ノーシェイム)。rulesに streak/streak_best/week_checked 列を追加(冪等ALTER)。写真=非永続(Phase5)・リアクション=非永続のまま。
- **ヘッダ/ヒーローの連続表示**: 現状は**自分ルールの連続(有言実行)を集約**(current=max(rule.streak)/best=max(streak_best))。usersに列は持たない。
- **【将来タスク】ヒーローの「連続記録」を運動ベースに切替**(実日付化とセット): 定義=「運動の有言実行連続」＝**予定日にdoneしたら+1／予定日にすっぽかしたら切れる／予定なし日は影響なし**(前回確定基準)。entriesの履歴から算出できる想定なので users 列は追加不要。実日付モデル導入時に実装。それまではルール連続を暫定表示(単位は「週」)。
